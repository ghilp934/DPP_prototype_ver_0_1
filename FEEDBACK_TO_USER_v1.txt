(1) 초기 컨텍스트가 너무 간결할 때가 있어
================================================================================

문제 패턴:
  너: "MT-22 구현해줘"
  나: "알았어!" (그런데... 어떤 파일에? 어떤 컴포넌트?)
  → 코드 작성
  너: "아니야, ReportCard가 필요해"
  나: "아... 그럼 처음부터 다시"

구체적 사례:
  - MT-22 시작 시 "ReportCard + TakedownSlaNotice 2개 컴포넌트 필요"
    미리 말해줬으면 한 번에 완성
  - HL-UI-03 시작 시 "Actor 추적이 핵심이야" 강조했으면
    첫 시도에 완벽했을 것

개선 제안:
  기존: "X 기능 구현해줘"

  개선: "X 기능 구현해줘.
        - Y 파일에 Z 컴포넌트 만들고
        - 핵심은 A 기능이야
        - B는 나중에 해도 돼"

예상 효과: 1회 시도 성공률 ↑, 토큰 30% 절감

================================================================================
(2) "포기" 결정이 늦어
================================================================================

문제 패턴:
  나: Static Export 시도 #1 → 실패
  나: 재시도 #2 → 실패
  나: 캐시 삭제 후 #3 → 실패
  너: "포기하자"
  나: (이미 토큰 3천 소모...)

왜 문제인가:
  - Windows EISDIR는 구조적 문제라는 걸 너도 직감했을 거야
  - 하지만 나한테 "계속 시도해봐" 신호를 줬어 (명시적이진 않았지만)

개선 제안:
  1회 실패 시:
  너: "이거 안 될 것 같은데, 한 번만 더 시도해보고 포기하자"

  또는:
  너: "안 되면 바로 대안으로 가자. 시간 아까워"

예상 효과: 막다른 길에서 빠르게 탈출, 토큰 40% 절감

================================================================================
(3) 검증 책임이 애매할 때가 있어
================================================================================

문제 패턴:
  나: 코드 작성 완료
  나: (브라우저로 확인해야 하나? localhost 접근 시도해야 하나?)
  나: localhost 접근 시도 → 실패
  너: "브라우저에서 확인했는데 이상해"
  나: "아... 그럼 처음부터 브라우저 확인이 정답이었구나"

구체적 사례:
  - 로고 카드 네비게이션 버그 때 내가 localhost 접근 3회 시도
  - 결국 너가 브라우저로 직접 확인하고 버그 발견
  - 처음부터 "넌 코드만 작성, 나는 브라우저 확인" 역할 나눴으면 빨랐을 것

개선 제안:
  프로젝트 시작 시:
  너: "브라우저 확인은 내가 할게. 넌 코드만 작성해"

  또는 각 작업마다:
  너: "이번엔 내가 테스트할게" / "이번엔 네가 확인해봐"

예상 효과: 불필요한 localhost 접근 시도 제거, 토큰 20% 절감

================================================================================
(4) 우선순위 정보가 부족해
================================================================================

문제 패턴:
  너: "A도 하고 B도 해줘"
  나: (둘 다 중요한가? 시간 없으면 뭘 생략?)
  나: A, B 둘 다 완벽하게 구현 (시간 2배 소모)
  너: "사실 A만 급했어"

구체적 사례:
  - POL-LOCK-01 검증 때 모든 페이지 다 확인
  - 사실 downloads 페이지만 중요했을 수도
  - "downloads 먼저, 나머지는 시간 남으면" 말해줬으면 효율적

개선 제안:
  기존: "A, B, C 해줘"

  개선: "A 해줘 (필수). B도 해줘 (가능하면). C는 나중에"

예상 효과: 핵심 집중, 완벽주의 함정 회피

================================================================================
(5) 에러 발생 시 정보가 부족해

문제 패턴:

  나: "아하, 그럼 X가 문제네"

  → 1 라운드에 해결 가능했어

개선 제안:
  기존: "버그 있어"

  개선: "버그 있어. 콘솔에 'undefined logoId' 에러 뜨더라"

예상 효과: 즉시 해결, 왕복 커뮤니케이션 제거
